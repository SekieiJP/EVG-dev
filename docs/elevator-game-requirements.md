# エレベーターゲーム 要件定義書（ドラフト v0.1）

## 1. 概要

### 1.1 ゲーム概要
会場のスクリーンとプレイヤーのスマホを用いた、対戦型Webパーティゲーム。プレイヤーはエレベーターの「乗る階」「降りる階」を予測しながら投票し、複数ステージにわたる合計得点で順位を競う。

### 1.2 想定環境
- **ホスト**: PCブラウザ（メインスクリーン投影＋サブスクリーン操作）
- **プレイヤー**: スマホブラウザ
- **同時接続上限**: 100名
- **ルーム**: 単一ルーム運用

### 1.3 用語定義
| 用語 | 定義 |
|------|------|
| ゲーム | 複数のステージで構成される一連の対戦 |
| ステージ | N, X, P, Q とイベント設定からなる1試行 |
| チケット購入 | 乗車階・降車階・予想イベント回答を確定する投票行為 |
| 乗車成功 | ステージ(4)の人数判定をパスしてエレベーターに乗れた状態 |
| 強制下車 | 定員超過で当該階の乗客・乗車希望者全員が降ろされる状態 |
| 棄権 | 制限時間内にチケット購入を完了できなかった状態（0点） |
| ホスト | 進行操作権限を持つ運営者 |
| スクリーン端末 | メインスクリーン投影画面を表示する端末（ホストと同一可） |
| セーブデータ | UUID単位で永続化される、複数ゲームをまたぐ戦績データ |

## 2. ゲームルール詳細

### 2.1 基本ルール
1. 1階からN階まで昇るエレベーターがある。定員はX名。
2. プレイヤーは乗りたい階と降りたい階をひとつずつ選ぶ。
3. エレベーターは各階に止まり、**乗車処理 → 人数判定 → 下車処理** の順に進める。
4. 乗車処理でX名を超えた場合、その階の乗客・乗車希望者**全員**が強制下車となる。エレベーターは乗客0人で次階へ。
5. 得点計算式:
   ```
   素点 = (実際の下車階 - 乗車階) × P - (申告下車階 - 申告乗車階) × Q
   最終得点 = 素点 + イベント加点・倍率による補正
   ```

### 2.2 入力制約
- 乗車階: 1〜N階の整数
- 降車階: 乗車階〜N階の整数（**下方向は不可**、**同一階は可**）
- 同一階指定の場合、乗車成功時は実上昇=1階（乗車階で人数判定通過→下車）と数える

> **補足**: 「同一階指定での実上昇=1階」は回答3の例（10階で乗車成功→同階下車=1階）に準拠する。

### 2.3 強制下車の挙動
- 同階で乗車申告したプレイヤーが複数いて、その合計が定員超過を引き起こす場合は**当該階の乗車希望者全員**がアウト。
- 強制下車に巻き込まれたプレイヤーの「実乗車階」は「申告乗車階」と同じ扱い（実上昇階数=0）。
- 強制下車後、エレベーターは乗客0で次階へ移動を継続する。

### 2.4 「乗車できなかった」場合の得点
- 強制下車・禁止階指定などで乗れなかった場合も、申告に対する**Q倍ペナルティ（チケット代金）は支払う**。
- 得点式: `0 × P - (申告下車階 - 申告乗車階) × Q`

## 3. イベント仕様

### 3.1 イベント種別
| ID | 名称 | 概要 | 設定パラメータ |
|----|------|------|----------------|
| E1 | 予想イベント | 試合中の事象を予想、正解で加点 | 問題文, 回答形式, 配点M, 不正解時点, 無回答時点 |
| E2 | 禁止イベント | S〜T階での乗降を禁止 | S, T |
| E3 | 倍率イベント | S〜T階での乗車成功点に倍率R | S, T, R |
| E4 | 特別階イベント | Y階で乗車成功していた場合に+M | Y, M |
| E5 | 人数指定イベント | A人以上乗車中の区間に倍率R | A, R |
| E6 | 眺望イベント | 乗車成功した場合のみ、降車階×M を加点 | M |

### 3.2 イベント適用ルール
- **倍率の合成**: 複数倍率が同区間に重なる場合は**乗算**で合成（例: R=2 と R=3 → 6倍）。
- **適用対象**: 倍率Rは**乗車成功点（P側）にのみ**適用。Q倍ペナルティには適用しない。
- **計算順序**: 乗車成功点 × 倍率合成値 → イベント加点を加算 → Q倍ペナルティを減算。
- **判定タイミング**: E4・E5は「2.1の(3)で乗車成功した瞬間に当該階にいた」かで判定。通過階での判定は行わない。

### 3.3 倍率イベントの粒度（2種類を独立実装）
- **E3a (区間倍率)**: S〜T階間の昇降階数のみに倍率を掛ける。
- **E3b (得点倍率)**: ステージの乗車成功点全体に倍率を掛ける（条件は別途指定）。

### 3.4 予想イベント（E1）詳細
- 1ステージあたり**最大2問**まで設定可能。
- 回答形式（4種類）:
  1. **整数入力**（例: 強制下車は何回発生する？）
  2. **範囲選択**（例: 合計乗車人数は○〜○人）
  3. **プレイヤー指名**（例: 最高得点者は誰？）
  4. **Yes/No**（例: 全員が乗車成功する？）
- 配点（正解時加点M、不正解時点、無回答時点）はイベントごとに設定。

### 3.5 禁止階のバリデーション方針
- UIでは送信前に**警告ダイアログを表示**。
- プレイヤーがOKを選択した場合は警告を無視して送信可能（戦略的に禁止階を選ぶ余地を残す）。
- 禁止階を指定して投票した場合、その投票は無効化され「乗車できなかった」扱い（Q倍ペナルティのみ発生）。

## 4. アーキテクチャ

### 4.1 全体構成
```
┌──────────────────┐
│  ホスト端末       │ ──┐
│  (操作画面)       │   │
└──────────────────┘   │   HTTPS    ┌────────────────────┐
                       ├──────────► │  GAS Web App        │
┌──────────────────┐   │   JSON     │  (doGet/doPost)     │
│ スクリーン端末    │ ──┤            │                     │
│ (投影画面+音声)   │   │            │  ┌───────────────┐  │
└──────────────────┘   │            │  │ スプレッドシート│  │
                       │            │  │ (永続化)       │  │
┌──────────────────┐   │            │  └───────────────┘  │
│  プレイヤー       │ ──┘            └────────────────────┘
│  スマホ           │
└──────────────────┘

* ホスト端末とスクリーン端末は同一端末でも別端末でも運用可能
* BGM/SE再生はスクリーン端末のみ
```

### 4.2 構成要素
| 要素 | 採用技術 | 役割 |
|------|----------|------|
| 配信 | 静的HTMLホスティング（GitHub Pages等） | クライアントSPA配信 |
| クライアント | HTML/CSS/JavaScript（フレームワーク選定要） | UI・アニメーション |
| バックエンド | Google Apps Script (Web App) | API・ゲーム進行管理 |
| データストア | Google Spreadsheet | 永続データ保存 |
| 通信方式 | HTTPSポーリング（2〜3秒間隔） | 状態同期 |

### 4.3 状態管理方針
- **Single Source of Truth は GAS** とする。
- クライアントはローカルでUIキャッシュを持つが、確定値はすべてGASから取得。
- ホストPCの一時切断時もGASに状態が残るため、再接続で復旧可能。
- プレイヤーはGASと通信できる限り投票継続可能。

### 4.4 GASレートリミット対策
- ポーリング間隔は **2〜3秒** を基準に調整。
- 1リクエストにステージ状態・残時間・自分の投票結果・ランキング差分をまとめて返す。
- 100人×2秒間隔 = 50req/sec が常時。GASの並列実行上限（30）を超える可能性があるため、**サーバ側でレスポンスをキャッシュ**し、リクエストが集中しても同一スナップショットを返す仕組みを検討。
- 後述のフェーズ別ポーリング戦略により、通信量を大幅に削減する。

### 4.4.1 フェーズ別ポーリング戦略（プレイヤー側）
ゲーム進行フェーズごとにポーリング要否を切り替えて通信量を削減する。

| フェーズ | ポーリング | 理由・備考 |
|----------|-----------|------------|
| タイトル | 不要 | ローカル状態のみ |
| 参加者受付中 | **要**（3秒間隔） | ゲーム開始検出のため |
| ステージ説明 | **要**（3秒間隔） | 次フェーズ開始検出のため |
| チケット購入受付中 | **要**（2〜3秒間隔） | 締切開始検出のため |
| 締切カウントダウン15秒 | **不要** | カウントダウンはローカル時計で進行。カウントダウン開始時に終了予定時刻が配信され、終了後に1回フェッチして集計フェーズへ移行 |
| 集計中 | **要**（短時間・1秒間隔） | 結果取得まで。サーバが演出開始時刻を返したら即座に演出フェーズへ |
| 結果発表アニメーション | **不要** | サーバ時刻同期されたローカル再生。Skip通知のみ別途検知（後述） |
| 中間ランキング表示 | **不要** | プレイヤーが「次のステージへ」ボタンを押すまで待機。ホストの口頭指示で各自が押下 |
| 最終結果表示 | **不要** | ゲーム終了 |

### 4.4.2 「次のステージへ」ボタンによる進行
- 中間ランキング表示後、各プレイヤー画面に「次のステージへ」ボタンを表示。
- ホストは口頭で「次のステージへ進んでください」と指示し、各プレイヤーが自身のボタンを押す。
- ボタン押下時に1回だけ状態フェッチを行い、サーバ側でホストが進行済みであれば次ステージへ遷移。
- ホスト未操作の場合は「ホストの操作待ち」を表示し、短時間ポーリング再開。
- 最終結果表示時は「最終結果を表示」ボタンを同じ仕組みで提供。

### 4.4.3 Skip検知と例外的ポーリング
- 結果発表アニメーション中はポーリングしないが、ホストのSkipに追従するため、**Skip検知のみ低頻度（5〜10秒間隔）でフェッチ**するか、アニメ終了予定時刻直前の最終確認フェッチで判定する。
- 詳細な実装方針は実装段階で決定。

### 4.4.4 ホスト側のポーリング戦略
- ホストは進行管理のため、プレイヤーよりも広いフェーズでポーリングを継続する。
- 締切カウントダウン中もロスタイム発生検知のためポーリング維持。
- 中間ランキング表示中も他ホストの操作検出のためポーリング維持。
- ただし結果発表アニメーション中は同様にポーリング不要。

### 4.5 時刻同期
- ゲーム読み込み時にクライアント→GASで時刻取得し、ローカル時刻との差分を保持。
- 「スタート時刻」をサーバ時刻で配布し、各クライアントは補正済みローカル時刻で同時にアニメ開始。

### 4.6 通信失敗時の挙動
- **送信失敗（タイムアウト等）**: クライアントは自動リトライ（指数バックオフ、最大3回）。
- **制限時間内に送信開始したリトライ**: 締切後でも、通常リトライ所要時間分の**ロスタイム**を設けて受信を認める。
- **制限時間内に送信できなかった**: 「棄権」を自動送信。
- **棄権送信も失敗**: ホスト側で**未着プレイヤーとして可視化**し、手動対応を促す。

### 4.7 セキュリティ・不正対策
| 対策 | 内容 |
|------|------|
| 締切後の改変防止 | サーバ側で締切時刻を検証、過ぎていれば拒否 |
| なりすまし防止 | UUIDによる本人確認（5.参照） |
| 投票改ざん防止 | サーバ側で入力範囲・禁止階等を再バリデーション |
| ランキング集計の信頼性 | クライアント計算結果は表示用、確定値はサーバで計算 |
| ホスト権限保護 | ホスト用URLへのアクセス時にパスワード認証 |
| スクリーンURL | 常時固定URL。投影専用のため認証なし（操作機能を持たない） |

## 5. プレイヤー識別とセッション

### 5.1 識別方式
- 参加時に**名前**を入力し、初回参加時にサーバ側で**UUID**を発行。
- UUIDはローカルストレージに保管。
- 名前の重複は不可（参加時に既存名と衝突したらリジェクト）。
- **統計集計はUUIDをキーとする**。名前は表示用属性であり、変更されても同一プレイヤーとして扱う。

### 5.2 再接続・引き継ぎ
- 同一ブラウザ: ローカルストレージのUUIDで自動再接続。
- 別ブラウザ・別端末: クライアントの**設定画面でUUIDを手動入力**することで引き継ぎ可能。
- 参加完了時に**自分のUUIDを画面表示**（コピーボタン付き）。

### 5.3 UUIDのエクスポート/インポート
- クライアントの設定画面に「データ引き継ぎ」機能を設置。
- **エクスポート**: 現在のUUIDをテキスト表示・コピー可能。
- **インポート**: 控えたUUIDまたは運営から提供されたUUIDを入力欄に貼り付けて引き継ぎ。
- Cookie/ローカルストレージが消失した場合のリカバリ手段として機能。
- 運営からのUUID提供フロー: ホストがサブスクリーンの参加者一覧から特定プレイヤーのUUIDを確認し、口頭・チャット等で本人に伝える。

### 5.4 名前変更
- クライアントの設定画面で名前変更可能。
- 変更は**次のステージから反映**。進行中のステージ表示は旧名のまま。
- 変更頻度の制限なし。**ステージ間に何度でも変更可能**。
- 重複チェックの対象は**セーブデータに存在する現行の全プレイヤー名**（過去ゲームの参加者を含む）。一度使われた名前は引退後も他人が使えない。
- 過去履歴の表示名は**変更時点の名前**を保持（履歴の整合性確保）。
- 統計集計はUUIDキーのため、名前変更によって過去戦績が分断されることはない。

### 5.5 ゲームをまたぐ統計
- セーブデータはUUID単位で永続化。
- 複数ゲームをまたいだ累積得点、参加回数、平均得点、最高得点等を集計可能。
- 過去ゲームの戦歴一覧をプレイヤー画面・ホスト画面の両方から閲覧可能。

## 6. ホスト機能

### 6.1 アクセス制御
- **ホスト用URL**: 操作機能を提供。アクセス時にパスワード認証を要求。
- **スクリーン用URL**: メインスクリーン投影画面のみを表示（**常時固定URL**、認証なし）。
- **ホストパスワード**: スプレッドシートの設定シートに記載し、**固定運用**。GASスクリプトプロパティではなく設定シートで管理することで、運営者が直接確認・変更可能。
- **同一端末運用**: 1台のPCで両URLを別タブ・別ウィンドウで開き、片方をプロジェクター出力に割り当てる。
- **別端末運用**: ホストはノートPC、スクリーンは会場PC等、独立した端末で運用可能。
- **同時ホスト**: 同一ホストURLに2名程度まで同時接続を許容。

### 6.1.1 同時ホストの操作競合
- 同一アクション（締切・次ステージ等）が同時発生した場合、**最初の操作のみ採用**し、後発は拒否。
- 他ホストの操作開始を検知した時点で、競合する操作ボタンを**ロック**して後発操作の発生を予防する。
- 他ホストが行った操作は、サブスクリーン上でリアルタイムに可視化（例: 操作ログ欄に「ホストB が締切を実行」など）。

### 6.2 サブスクリーン（ホスト操作用）
#### 6.2.1 ゲーム設定の事前作成GUI
- 複数ステージを順序付きで作成。
- 各ステージに N, X, P, Q とイベントを設定。
- イベントの追加・並べ替え・削除。

#### 6.2.2 設定のインポート/エクスポート
- **エクスポート**: JSON形式で全設定をダウンロード。
- **インポート**: JSONをアップロードして読み込み。
- **部分インポート**: 個別ステージのみのインポートを可能とする。
- **スキーマバージョン**: JSONに `schemaVersion` を含め、将来の互換性を担保。

#### 6.2.3 進行管理
- 参加者一覧と現在の入力状況（投票済み・未投票・通信エラー）。
- 各参加者のUUID表示（リカバリ提供用）。
- リアルタイムランキング表示。
- 締切ボタン、次ステージへの遷移ボタン、結果発表トリガ、アニメーションSkipボタン。

#### 6.2.4 過去戦歴の参照
- UUID単位の累積戦歴閲覧。
- 過去ゲームの結果サマリと、特定プレイヤーの履歴抽出。

### 6.3 スクリーン端末（投影用）
#### 6.3.1 ゲーム進行フェーズ表示
| フェーズ | 画面内容 |
|----------|----------|
| 参加者受付中 | ルーム参加URL/QRコード、参加者リスト |
| ステージ説明 | N, X, P, Q とイベント詳細 |
| チケット購入受付中 | 投票進捗、参加者数 |
| 締切カウントダウン | 15秒のカウントダウン演出 |
| ステージ結果発表 | エレベーター上昇アニメーション |
| 中間ランキング | 累積順位とスコア差 |
| 最終結果 | 総合順位・表彰演出 |

#### 6.3.2 結果発表アニメーション
- エレベーターが1階からN階へ上昇する演出。
- 各階での乗車・下車・強制下車をビジュアル化。
- プレイヤースマホ側演出と**ローカル補正時刻で同期再生**。
- ホスト操作画面から**Skipボタン**で演出をスキップ可能（**ホストのみ操作可**）。Skip時はクライアント側演出も同期して終了状態へ遷移。
- Skip時のBGM/SEは**フェードアウト**で停止。即時停止はしない。
- 演出時間は実装後の調整事項。

#### 6.3.3 BGM/SE再生
- **再生はスクリーン端末のみ**。プレイヤースマホでは音声を一切再生しない。
- 音声リソースは**静的配信元に同梱**（GitHub Pages等のリポジトリに配置）。
- フェーズごとに対応するファイル名規約（例: `bgm_voting.mp3`, `se_force_off.wav`）で配置。
- フェーズと音声ファイルの対応一覧は今後別途定義。
- ホストはサブスクリーンから音量調整・ミュート操作が可能。
- 音声ファイル差し替え機能は**初版では不要**（将来の検討事項）。

#### 6.3.4 スクリーン端末の起動
- スクリーン端末ではブラウザの自動再生制限により、初回操作なしに音声を再生できない。
- スクリーン用URLにアクセスした時点で、画面上に「準備完了」ボタンを表示し、**スクリーン端末側でクリック操作を要求**。
- クリックでオーディオコンテキストが起動し、以後ホスト操作によって音声再生可能。
- **再接続時**: 切断・再接続後も音声再生のため、再度「準備完了」ボタンのクリックを要求する。

### 6.4 締切処理
- ホストが「締切」ボタン押下後、スクリーンで**15秒カウントダウン**。
- カウントダウン中もプレイヤーは投票変更可能。
- 0秒時点で締切確定、以降の投票は拒否（ロスタイム除く）。
- スマホ画面でも締切状況がリアルタイムでわかるよう表示。

## 7. プレイヤー機能

### 7.1 参加登録
- 名前入力（重複不可）。
- UUID発行・ローカルストレージ保存。
- 過去UUIDからの引き継ぎ入力欄。

### 7.2 チケット購入

- 自分の過去ステージ・過去ゲームの履歴。

## 10.5 成績指標「Skill」

### 10.5.1 標準化得点
Skill計算ではプレイヤーの素点をそのまま使わず、ステージごとに**標準化得点**へ変換した値を用いる。

- 当該ステージ参加者の最低点を10、最高点を100に線形変換。
- **全員同点の場合は全員を55として扱う**。
- **参加者が1名のみの場合は標準化不能のため、当該ステージのSkill値は記録しない**。

```
StandardizedScore = 10 + (rawScore - minScore) / (maxScore - minScore) × 90
（minScore == maxScore のとき: 全員 55、参加者1名のとき: 算出しない）
```

### 10.5.2 ステージ別Skill値
各ステージ終了時、参加した全プレイヤーに対してSkill値を計算する。

```
StageSkill = (上昇成功階数 / (ステージ階数 × 定員 / 参加人数)) × 60
           + (標準化得点 / (参加者の標準化得点合計 / 参加人数)) × 40
```

- 計算結果が0未満になる場合は0にクランプ（最低値 = 0）。
- `参加者の標準化得点合計 == 0` の場合（参加人数0など特殊状況）は**第2項を0**とする。

| 変数 | 定義 |
|------|------|
| 上昇成功階数 | 当該ステージで実際に上昇した階数（強制下車・不参加・乗車失敗時は0） |
| ステージ階数 | ステージ設定のN |
| 定員 | ステージ設定のX |
| 参加人数 | 当該ステージで**乗車希望または予想回答のいずれかを送信したプレイヤー数**（完全棄権者は除外） |
| 標準化得点 | 10.5.1で算出した値 |

### 10.5.3 現在Skill値
プレイヤーが過去に参加した**全ステージ**のStageSkill値のうち、上位 **2,3,4,5番目** の4つの値の合計。

- 最高値（1位）は集計から除外する設計意図: **奇跡の1ステージを軽視**する。
- 参加ステージ数が5未満の場合は、不足分を0として加算する。
- 計算ロジックは今後変更の可能性あり。

### 10.5.4 表示する指標
プレイヤーの累積戦歴ページに以下を表示する。スマホ画面では優先度の高い指標を上部に固定し、残りは折りたたみで表示する。

| 指標 | 内容 |
|------|------|
| 現在Skill値 | 10.5.3の定義 |
| 平均Skill値 | 全ステージのStageSkillの算術平均 |
| 合計Skill値 | 全ステージのStageSkillの総和 |
| 累積得点 | 全ゲームの最終得点合計 |
| 平均得点 | ステージあたりの平均得点 |
| 最高得点 | ステージあたりの最高得点 |
| 参加ゲーム数 | 参加したゲーム数 |
| 参加ステージ数 | 参加したステージ数 |
| 強制下車回数 | 強制下車に遭遇した回数（自爆・巻き込まれを区別せず合算） |
| 予想イベント正解率 | 正解数 / **回答した予想イベント数**（無回答は分母に含まない） |
| 優勝回数 | ゲーム終了時に1位だった回数（参加人数による重み付けなし） |
| 表彰台回数 | ゲーム終了時に1〜3位だった回数（参加人数による重み付けなし） |

### 10.5.5 集計タイミング
- 各指標はゲーム終了時に集計して `save_data` に保存（事前集計方式）。
- 個別ステージの結果は `stage_results` に保存し、Skill値の再計算に備える。
- 戦歴閲覧時は保存済み値を読み出すのみ。
- 指標定義が変わった場合は、**ホストが管理画面から「再集計」操作を手動実行**することで全データを再計算する。スプレッドシート上の関数や再集計スクリプトで処理。

### 10.5.6 現在Skill値の配信と表示
- **表示位置**: タイトル画面および戦績ページにおいて、各プレイヤーの現在Skill値を表示する。
- **ゲーム中の表示**: ゲーム進行中は他プレイヤーのSkill値を参照しない（通信節約のため）。自分自身のSkill値のみ画面に表示する。
- **配信タイミング**: タイトル画面表示時にGASから戦績データを取得しローカルストレージにキャッシュ。`/api/player/join` のレスポンスに自分のSkill値を含めて返却。
- **更新タイミング**: ゲーム中はSkill値を更新しない。ゲーム終了後、次回タイトル画面表示時に最新値を取得。
- **表示形式**: 小数2桁で表示。新規プレイヤー（参加ステージ数0）は `0.00`。
- **Skill値が計算されないステージ**: 参加人数1名のステージや全員同点でないが標準化不可能なエッジケースでは、当該ステージのStageSkillを記録しない。

## 11. 非機能要件

### 11.1 パフォーマンス
- ポーリング応答: 平均1秒以内。
- 100人同時アクセスでもGASのレートリミット内に収まる設計。

### 11.2 可用性
- ホスト・プレイヤーともに再接続可能。
- GAS障害時はクライアントにエラー表示しリトライ。

### 11.3 ブラウザ対応
- スマホ: iOS Safari 最新2バージョン、Android Chrome 最新2バージョン。
- PC: Chrome / Edge / Firefox / Safari 最新版。
- スクリーン端末: 上記PCブラウザに加え、HTML5 Audio APIが動作すること。

### 11.4 アクセシビリティ
- 色覚多様性に配慮した配色。
- 大きめのタップターゲット（最小44×44px）。

### 11.5 国際化
- 当面は日本語のみ。文言は別ファイル化して将来の多言語対応に備える。

### 11.6 ロギング・運用支援
不具合検証および運用のため、以下のログ・データ可視化を行う。

#### クライアント側
- **記録対象**: 画面上の情報に影響するすべての通信（リクエスト/レスポンス、ステート変更、エラー、リトライ）を記録。
- **保持件数**: ローカルストレージに直近一定件数を保持（具体的件数は実装時に決定）。
- **閲覧**: 設定画面の「通信ログ表示」から閲覧可能。
- **サーバ送信は行わない**。クライアントログはローカル閲覧のみとする。クライアント発の通信失敗はGAS側に到達しないため、運営側はクライアント側のログを直接参照できない点に留意。

#### GAS側
- 全APIアクセスをGAS実行ログ（`console.log`/`Logger.log`）に出力（タイムスタンプ、エンドポイント、UUID、結果ステータス）。
- エラー発生時はスタックトレースも出力。
- GASの実行ログUIから直接確認可能とする。シートとしては保存しない。

#### スプレッドシート構成
最低限、以下のシートで運用状況を可視化する。スプレッドシート自体はホストのみがアクセスする前提のため、シート保護は不要。

| シート名 | 内容 | 主キー |
|----------|------|--------|
| `config` | ホストパスワード、運用設定 | - |
| `save_data` | プレイヤー単位×ゲーム単位の戦績データ（参加時点の表示名スナップショット含む、事前集計指標含む） | (UUID, gameId) |
| `stage_results` | プレイヤー単位×ステージ単位の結果（StageSkill値含む、Skill再計算用） | (UUID, gameId, stageId) |
| `players` | プレイヤーマスタ（UUID、現在の表示名） | UUID |
| `current_game` | 現在進行中のゲーム状況（参加者・投票・スコア） | - |
| `stage_settings` | 現在および過去のステージ設定 | (gameId, stageId) |
| `game_history` | 過去ゲームのサマリ | gameId |

### 11.7 データ保存方針
- セーブデータは複数ゲームをまたいで永続化。
- 1ゲーム終了時に `current_game` の内容を `game_history` に転記し、UUID×ゲーム別の戦績を `save_data` に追記。
- `save_data` には参加時点の表示名をスナップショットとして保存（履歴の整合性確保）。
- `players` シートは現在の表示名のみ管理し、変更時は上書き。
- 累積戦歴は `save_data` からUUID単位で集計して算出。
- スプレッドシート行数の上限（500万セル）に達する前に、古いデータのアーカイブ運用が必要（12章参照）。

### 11.8 ゲームID
- **採番**: ホストが設定作成時に**任意に命名**（例: 「新年会2026」）。
- **重複時**: セーブデータに同名のゲームIDが既に存在する場合、**通し番号連番を付加**（例: 「新年会2026_2」「新年会2026_3」）。
- ゲームIDは `save_data`、`stage_settings`、`game_history` の主キーまたは外部キーとして使用。

### 11.9 戦歴の公開範囲と表示方針
- 過去ゲーム・ステージの**結果（参加者一覧と各々の得点・順位）は全プレイヤーに公開**。プレイヤー画面・ホスト画面いずれからもゲーム単位・ステージ単位で結果閲覧が可能。
- **プレイヤー個人にフォーカスした戦歴ページは自分自身のものしか表示できない**。他人の名前をクリックして個人ページを開く機能は提供しない。
- 現在Skill値は全員分が**タイトル画面の戦績表示**で確認可能（プレイヤー一覧形式）。
- ゲーム進行中は他プレイヤーのSkill値や統計を画面に表示しない（通信節約のため）。
- 戦歴ページのパフォーマンス対応: 初期表示は集計済みサマリ、詳細はオンデマンド取得、12項目の指標は折りたたみで表示。

## 12. 残課題・今後の検討事項

1. リアルタイム性が不足する場合のWebSocket/SSE代替案検討。
2. 結果発表アニメーションの演出時間の最終決定（実装後の調整事項）。
3. Skip時のBGM/SEフェードアウト時間の決定。
4. フェーズと音声ファイルの対応一覧の作成（BGM・SEそれぞれ）。
5. クライアントログのローカル保持件数の決定。
6. 大規模イベントでの負荷試験計画。
7. スプレッドシート容量上限到達時のアーカイブ運用フロー。
8. 静的配信元の音声ファイル合計サイズと配信レイテンシの検証（GitHub Pagesの1GB制限・100MB/ファイル制限）。
9. `save_data` および `stage_results` の具体的なカラム構造（事前集計指標のカラム名・データ型）の確定。
10. Skip検知の実装方式: 低頻度ポーリング vs アニメ終了直前の最終確認フェッチ、どちらを採用するか。
11. プレイヤーが「次のステージへ」ボタンを押した際、ホスト未操作の場合の待機UI仕様（ポーリング再開頻度・タイムアウト表示）。
12. 離脱者の取り扱い: 「次のステージへ」ボタンを押さないまま次ステージが始まった場合、当該プレイヤーは自動的に次ステージへ遷移するか、置き去りにするか。
13. 「再集計」操作の対象範囲: 全データを一括再計算するか、特定ゲームのみを選んで再計算可能とするか。
