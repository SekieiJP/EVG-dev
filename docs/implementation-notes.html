デザインの決定

  • 初版はフレームワークなしのHTML/CSS/JavaScriptにした。要件ではフレームワーク選定が未確定だったため、GitHub PagesやGAS連携に載せやすい依存なし構成を優先した。
  • ローカル確認用に、ブラウザのlocalStorageを使う単一ルーム実装を入れた。GAS未デプロイでも画面遷移と集計を確認できるようにするため。
  • ホスト・スクリーン・プレイヤーは同じHTMLを?view=hostなどのクエリで切り替える方式にした。静的配信ファイル数を増やさず、同一端末・別端末運用の両方に対応しやすい。
  • ホストパスワードの既定値はhostにした。GASではconfigシートのhostPasswordで変更可能。
  • クライアント通信ログの保持件数は80件にした。要件では件数未定だったため、スマホで過剰にならない範囲の暫定値とした。
  • 結果発表アニメーションはCSSのエレベーター上昇とランキング表示で実装した。各階の詳細な乗降者表示はタイムラインデータから生成し、到達済み階だけを可視化する。
  • 結果発表画面は投影先が横画面である前提を明示し、固定かごを中心にフロアがスクロールする表現にした。階数、かご、プレイヤー状態は列を分けて重なりを避ける。
  • 登録済みユーザーは、ローカルストレージのUUIDが現在ルーム内のプレイヤーと一致する場合に限って氏名入力画面を経由しない。UUIDが現在ルームに存在しない場合は、新規参加または折りたたみ内からの明示的な引き継ぎ操作を求める。
  • 乗降時の一時停止は、CSSのstepsアニメーションで各階ごとに停止している。厳密に「イベントがある階だけ1秒停止」ではなく、暫定的に各階単位の停止として実装した。
  • 結果発表中のスコア更新では、予想イベントの得点はすべての結果が確定する最上階到達時に加算する扱いにした。
  • QRコード生成は外部CDNではなく、MITライセンスのqrcode-generator@1.4.4をリポジトリに同梱する選択にした。会場ネットワークに依存しないため。
  • GAS接続先はランタイム入力ではなくビルド時定数にした。運営中に誤ったURLを入力するリスクを避けるため。
  • GASへのPOSTはapplication/jsonではなくtext/plainでJSON文字列を送る。Apps Script Web AppでCORSプリフライトに詰まりにくい運用を優先した。
  • GASホスト認証は、静的クライアントに埋め込むapiKeyと、運営者が入力するhostPasswordの二段構成にした。認証後のhostTokenはCacheServiceに保存するランダムトークンで、推測可能な自己記述トークンにはしない。
  • apiKeyはデプロイIDと同値であることを正とし、過去セットアップで作られたevg-形式の値がconfigシートに残っていてもデプロイIDを受理する。再度setupElevatorGameSheets()を実行した場合はevg-形式をデプロイIDへ移行する。
  • Hostブラウザ集計commitは、運営用Hostを信頼する設計に寄せる判断をした。不正耐性よりGAS実行時間削減を優先し、最低限の防御としてhostToken、roomVersion、ステージ一致、二重集計拒否を入れる。
  • Apps Scriptエディタに直接配置するファイルはCode.gsだけにし、appsscript.jsonはclasp等で使う任意のmanifestとして扱う。
  • topPlayer予想の正解は、予想イベント得点を加える前のステージ得点1位にした。予想得点によって正解が変わる自己参照を避けるため。
  • 予想イベントでmetricとcorrectAnswerが両方指定された場合は、ゲーム結果から得られるmetricを正解にする。correctAnswerは固定正解や暫定設定用のフォールバックに限定した。
  • 複数プレイヤーブラウザハーネスでは、testSlotクエリをUUID保存キーのサフィックスとして使う。同一ブラウザ内でもプレイヤーごとのUUIDを分離して、実端末を複数用意しなくても参加フローを確認できるようにした。
  • 次ゲームへの移行は、前ゲームの結果とプレイヤー情報を戦歴・セーブデータへ退避したうえで、現在ゲームの表示参加者は空から始めることにした。各プレイヤーはアクセスまたは復元操作後に現在ゲームへ表示される。
  • Playerの中間ランキング保持はlocalStorage内のスナップショットで実装した。ホストが次ステージへ進めても、本人が次へを押すまでPlayerタブはランキングのまま残る。
  • 事前登録する次ゲーム候補はstage_settingsではなくgame_configsに分離した。stage_settingsは実際に走ったゲームのステージ設定履歴、game_configsは今後開始するテンプレートという役割にする。
  • game_configsの行は使用後もACTIVEのまま残す。テンプレートを繰り返し使う運用を優先し、不要な候補はSpreadsheet上でARCHIVEDへ手動変更する。
  • 中断して次ゲームへ進む場合、保存対象は集計済みステージだけにした。未集計の投票や受付中ステージは確定結果ではないため、履歴には残さず破棄する。
  • 音声ファイル名はすべて英小文字とアンダースコアのmp3に固定した。BGMはbgm_フェーズ名.mp3、SEはse_イベント名.mp3の形式にした。
  • se_ride.mp3は、結果発表で1人以上が乗車中、かつその階で新規乗車・下車・強制下車が発生しない場合の上昇演出として扱う。
  • Firebase Spark初期版では、静的SPA配信は維持し、ゲーム中のSingle Source of TruthをRealtime Databaseへ移す計画にした。GitHub Pages配信を残すことで、配信面の変更を最小化し、通信層の差し替えに集中するため。
  • Firebase版のデータ構造は、現行のroom全体JSON保存ではなく、public、players、tickets、ticketPresence、results、scoresへ分割する方針にした。Sparkの10GB/月ダウンロード枠に対して、全room再配信を避けるため。
  • Firebase SparkではCloud Functionsを前提にしないため、初期移行ではHostブラウザ集計を継続し、Realtime Database Rulesとtransactionで最低限の権限・二重操作対策を行う計画にした。
  • Firebase Authは匿名ログインを使う計画にした。既存UUIDはAuth uidへ完全置換せず、プレイヤー復旧や表示用のゲーム内IDとして移行互換を検討する。
  • Firebase実接続に使うWeb API key、authDomain、databaseURLはgame/assets/js/config.jsに置いた。Firebase Web API keyは公開前提の識別子であり、秘匿情報として扱わない。
  • Firebase adapterは既存UIとの接続面を小さくするため、画面側にはGAS互換APIを見せ、内部だけをRealtime Databaseの分割ノードへ変換する設計にした。
  • Firebaseの匿名Authは、既存セッションが復元されるまで最大1秒待ってから新規匿名ログインする。ホスト認証済み端末のuidがリロードごとに変わると、hostTokenとDB上のhostUidがずれるため。
  • Firebase版の状態反映ではroomVersion比較で古い通知を捨てないことにした。Playerの復元・参加など自分だけの子ノード更新ではpublic側のversionを進めないため、比較するとHostのフェーズ通知を誤って捨てる可能性がある。
  • GASのplayersシートは現在ゲーム参加者だけで上書きせず、既存UUIDを保持したまま現在参加者をマージする。過去UUID復元の起点を消さないため。
  • ルール説明スライドは、公開用game/配下に単体HTMLとして置いた。ゲーム本体のSPAや依存ライブラリへ組み込まず、会場でURLを直接開ける簡易資料として扱うため。
  • ルール説明スライドの図は、乗車処理、定員超過、得点式、役割進行を示す模式図に限定した。高層ビルなどのモチーフイラストは、理解すべきルール情報を増やさないため使わない。
  • ステージ制作支援ツールは、既存アプリと同じschemaVersion、gameMeta、stages[].params、stages[].events形式を出力する単一HTMLにした。GitHub Pagesやローカルファイルでそのまま開けることを優先した。
  • JSON書式変更に備え、制作ツールにはトップレベル、ステージ、イベントごとの「追加JSON」欄を置いた。GUIがまだ知らないキーはここで保持・出力できるようにする。
  • 予想イベントの範囲選択肢はranges、options、choicesから出力キーを選べるようにした。既存エンジンが3種類を受け付けるため、Spreadsheet運用側の表記に合わせられるようにする。
  • 次段階の目標構成では、既存UIやengine.js互換よりもRTDBの小ノード購読、Host role allowlist、public単位のフェーズtransactionを優先することにした。既存のGAS互換adapterは移行足場として扱い、最終形では画面別command/query APIへ置き換える。
  • Host認証画面にもinternal-statusを表示し、未認証状態でFirebase uidを確認できるようにした。allowlist登録に必要な値を画面上で取得できるようにするため。

仕様解釈

  • 強制下車に巻き込まれた既存乗客は、強制下車前に人数判定を通過済みの階数分を上昇報酬の対象にする。当該階で新たに乗ろうとして強制下車になったプレイヤーは成功階数0階とする。
  • E3aの区間倍率は、人数判定通過後に乗車中だった成功階がイベントのS〜T内にある場合に適用する。例: 7〜10階は7階、8階、9階、10階の成功階が対象。
  • E3bの得点倍率は、申告乗車階から申告降車階までの経路がS〜Tに触れる場合、乗車成功点全体へ適用する。要件の「条件は別途指定」が未確定のため、設定JSONにあるfromFloor/toFloorを条件にした。
  • E5は、人数判定通過後に乗車中人数がしきい値以上だった成功階の乗車成功点へ倍率を掛ける。
  • E4は、人数判定通過時にその階にいたプレイヤーへ加点する。ブラウザエンジン・GAS集計とも同じ判定にしている。
  • 同一階指定は、人数判定通過後に即時下車し、成功階数1階として扱う。運賃も申告階数1階分を支払う。
  • 範囲選択の予想は、選択肢のvalueを回答として保存し、正解メトリクスの数値が選択肢のmin/max内なら正解とする。片側だけ指定した範囲も許容する。
  • プレイヤー指名の予想は、表示名ではなくUUIDを回答として保存する。名前変更があっても判定対象を安定させるため。
  • 名前重複は現行ルーム内のプレイヤーで判定している。今後GAS接続時は、UUIDと現在名の対応を管理するplayersまたはsave_data由来の現在名で重複判定する。
  • GASのreveal-resultは、すでに当該ステージの結果が存在する場合は拒否する。再集計は別APIのrecalculateに限定し、通常進行で累積得点とSkill履歴が二重加算されないことを優先した。
  • 完了済みゲームのローカル戦歴はcompletedGamesに退避する。次ゲームのゲーム内スコアは0から始めるが、History画面では退避済み結果と現在ゲーム結果を合算して表示する。
  • 次ゲーム開始時は、現在ゲームのステージ結果が存在する場合にsave_data、stage_results、game_historyへの保存を先に試みる。既存行がある場合は重複保存しない。
  • 中断保存されたゲームのサマリにはinterrupted=trueと中断時のfinalPhaseを記録する。ランキングや個人戦歴では、集計済みステージだけを含む短縮ゲームとして扱う。
  • Screen音声は準備完了操作後に再生を試みる。ブラウザの自動再生制限や未配置ファイルで失敗した場合は、クライアント通信ログへ記録し、進行自体は止めない。
  • Playerの結果表示解禁は、スクリーン演出のanimationStartedAtとステージ階数から計算した最終階到達時刻で判定する。ホストのフェーズ遷移ではなく演出完了時刻に合わせる。
  • GASのsave_dataに保存する12指標はゲーム終了時点の事前集計値とした。個別ステージ再計算に必要なStageSkillと結果詳細はstage_resultsへ保存する。
  • GASのPlayer向け状態レスポンスは、UI表示に必要な参加者名と本人チケットを優先し、全チケットや発表前結果は返さない。Screen向け状態は投影表示のため全体チケットと結果を返す。
  • 制作ツールのステージ単体インポートは、既存HostのStage Importと同じく現在の末尾へ追加する挙動にした。差し替えや挿入位置指定は、追加後の並べ替えで対応する。
  • 制作ツールの検証では、E1予想イベントは仕様通り1ステージ最大2問をエラー扱いにした。一方、イベント総数は現行エンジンが先頭10件に丸めるため、エラーではなく警告にしてJSON出力自体は可能にした。
  • 2026-06-02追記: 締切後のHost画面はRTDB上のフェーズ自動書き換えではなく、countdownEndsAt/tallyingEndsAtを使ったローカル時刻表示として扱う。Hostも1秒tickで再描画し、移動完了後に次へを押せるようにした。
  • 2026-06-02追記: Firebase RTDBから返る配列はオブジェクト形になることがあるため、stageResultsとcompletedGames復元時にtimeline、各階イベント内の乗車リスト、rankings、各プレイヤーの内訳配列を正規化する。
  • 2026-06-03追記: Host画面が開いていて認証済みの場合、移動完了後に手動の次へと同じcommitHostTally()経路を自動起動する。失敗時は10秒後に再試行し、同一ステージの二重集計は既存結果の存在とキーで抑止する。
  • 2026-06-03追記: Screen画面のステージ説明フェーズでは、購入受付前で誤解を招くためチケット購入人数の円グラフを表示しない。投票受付中以降は従来通り表示する。

  • 2026-06-05追記: 同じゲームJSONで再開始できるようにした。gameIdはタイトル+日時+短いランダム値で生成し、同一JSONを同日に複数回開始しても衝突しにくい形にした。
  • 2026-06-05追記: ゲーム説明用のHTMLスライドgame/rule-slides.htmlを追加した。基本ルール、チケット、強制下車、得点式、イベント、進行役割を投影向けに説明する。

逸脱

  • GAS通信はUSE_GAS_API有効時のfetch接続、hostToken連携、serverTime補正まで実装した。ただし、実GASデプロイURLでの現地疎通、同時接続負荷、CORS挙動のブラウザ横断確認は未実施。
  • BGM/SEファイルは同梱していない。音声ファイル名規約が未確定で、素材も未提供のため、スクリーンの初回「準備完了」とAudioContext起動のみ実装した。
  • GASのcurrent_gameは完全正規化ではなく、状態JSONを複数行チャンクで保存する実装にした。初版の整合性と競合制御を優先しつつ、1セル上限を避けるため。
  • 結果発表中のフェーズ更新検知は、GAS通信中は低頻度フェッチと最上階到達時の最終確認として残している。Firebase通信中はRTDB購読でフェーズ更新を受け取る。
  • 同時ホストのボタンロックは、GASのLockServiceによる先着処理を優先し、UI上のリアルタイムロック表示は未実装。
  • game_configs追加後の既存Spreadsheetでは、再度setupElevatorGameSheets()を実行して新シートと初期サンプル行を作る必要がある。
  • 結果発表終了時のBGM/SEフェードアウトは未実装。現状はフェーズBGM切替とSE発火トリガーの準備までで、フェード時間は引き続き未決定。
  • 制作ツールはゲーム進行中のHost画面には組み込まず、tool/stage-creator.htmlとして独立させた。Host画面上での直接編集は未実装。
  • Firebase通信層は、既存画面との互換を優先してGAS API互換のadapterとして実装している。購読は画面別に分けたが、書き込みAPIの形はGAS互換を維持している。
  • Firebase adapterはsnapshotノードを新規保存しない実装へ変更し、購読も画面別の子ノード購読へ移行した。ただしHost操作の書き込みは既存エンジン互換を優先し、room単位transactionを継続している。
  • Firebase版ではScreenの「同一端末同期」ボタンを表示しない。RTDB購読が主経路のため、同一端末BroadcastChannel/localStorage同期で別画面の古い状態を取り込むリスクを避ける。
  • GASモードとlocalStorage進行モードは実行経路から外した。過去実装やテスト資産は残すが、画面の通常操作はFirebase adapter経由だけにする。
  • Firebase版ではSpreadsheetのgame_configs候補読み込みを行わない。次ゲームは当面Host画面の設定JSON Importで開始する。
  • Firebase Security Rulesは、Host uidだけがroom全体を書けるようにし、Playerは自分の参加者ノードとチケットノードだけを書けるようにした。ただし、Cloud Functionsなしではホストパスワード検証やHost再取得の正当性、複雑なゲームルール検証をRulesだけで表現しきれない。
  • Firebase Spark前提では、サーバ側Cloud Functionsによる集計検証を意図的に先送りする。Sparkで開始するための制約だが、Hostブラウザを信頼する現行GAS版と同程度の信頼モデルが残る。
  • 目標アーキテクチャ文書では、現行コードに残るroot room materialization、root room transaction、GAS/local進行分岐を廃止対象として明記した。現時点の実装とは一致しないが、次のリファクタリングで解消する前提の意図的な設計差分である。
  • UI内部は引き続きstate.room形式のViewModelを使う。RTDBの通常読み書きは小ノード化したが、既存描画コードを一度に全面分解すると検証範囲が大きすぎるため、UI境界の完全なViewModel分離は次の整理対象として残す。
  • 現行engineのフェーズ名tallyingと時刻名tallyingEndsAtは残した。目標文書のmoving/movingEndsAtへ改名するにはengine、UI、既存テスト、保存済みデータの同時移行が必要なため。

トレードオフ

  • GASシートを完全正規化する案もあったが、進行中状態はチャンク化JSONスナップショット方式にした。理由は、複雑なステージ設定・投票・演出状態を一括でロック更新でき、同時ホスト競合を扱いやすいため。履歴とマスタはsave_data等へ分割保存する。
  • クライアントでWebSocket相当のリアルタイム通信層まで作る案もあったが、GAS前提ではポーリングを選んだ。参加者100人規模では、要件通りフェーズ別ポーリング間隔の調整とサーバ側キャッシュが次の負荷対策になる。
  • 登録前トップ画面でもポーリングして即時にゲーム進行へ追従する案は、Apps Scriptの実行時間上限に対して費用が大きいため採用しない。参加済み端末、認証済みHost、表示開始済みScreenだけが低頻度で状態取得する。
  • 快適度への影響として、Playerのフェーズ追従は自動即時ではなくなる。投票中の締切検知は最大10秒遅れ、ランキング後の次フェーズ追従は「次へ」押下時確認になる。一方、同一端末ScreenはGAS待ちがなくなり反応が良くなる。
  • 戦績キャッシュは通信量削減を優先し、同一ステージ内では古いGAS戦績を再取得しない。現在進行中ステージのローカルroom由来の表示は引き続き即時反映する。
  • Host自動集計は、認証済みHost画面が開かれていることを前提にした。Host端末が閉じている場合は、別端末Screen/Playerだけでは集計commitを開始しない。
  • GAS通信リトライは0.4秒、0.8秒程度の短い遅延に限定した。長いリトライはApps Script実行時間節約と利用者操作の明快さを損なうため採用しない。
  • ランキング保持はGASに既読状態を保存せず、Player端末ごとのローカル状態に限定した。通信量とサーバ状態の複雑化を抑えるため。
  • Playerの最終結果後の次ゲーム検知は自動ポーリングではなくボタン取得にした。切り替わりの即時性は落ちるが、ゲーム終了後に参加者端末が放置されてもGAS実行時間を消費し続けない。
  • 中断操作を許すと誤操作リスクが上がるため、Host画面では中断して開始という別ラベルと確認ダイアログで、未集計データが破棄されることを明示する。
