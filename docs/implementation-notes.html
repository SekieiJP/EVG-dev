デザインの決定

  • 初版はフレームワークなしのHTML/CSS/JavaScriptにした。要件ではフレームワーク選定が未確定だったため、GitHub PagesやGAS連携に載せやすい依存なし構成を優先した。
  • ローカル確認用に、ブラウザのlocalStorageを使う単一ルーム実装を入れた。GAS未デプロイでも画面遷移と集計を確認できるようにするため。
  • ホスト・スクリーン・プレイヤーは同じHTMLを?view=hostなどのクエリで切り替える方式にした。静的配信ファイル数を増やさず、同一端末・別端末運用の両方に対応しやすい。
  • ホストパスワードの既定値はhostにした。GASではconfigシートのhostPasswordで変更可能。
  • クライアント通信ログの保持件数は80件にした。要件では件数未定だったため、スマホで過剰にならない範囲の暫定値とした。
  • 結果発表アニメーションはCSSのエレベーター上昇とランキング表示で実装した。各階の詳細な乗降者表示はタイムラインデータから生成し、到達済み階だけを可視化する。
  • 結果発表画面は投影先が横画面である前提を明示し、固定かごを中心にフロアがスクロールする表現にした。階数、かご、プレイヤー状態は列を分けて重なりを避ける。
  • 登録済みユーザーは、ローカルストレージのUUIDが現在ルーム内のプレイヤーと一致する場合に限って氏名入力画面を経由しない。UUIDが現在ルームに存在しない場合は、新規参加または折りたたみ内からの明示的な引き継ぎ操作を求める。
  • 乗降時の一時停止は、CSSのstepsアニメーションで各階ごとに停止している。厳密に「イベントがある階だけ1秒停止」ではなく、暫定的に各階単位の停止として実装した。
  • 結果発表中のスコア更新では、予想イベントの得点はすべての結果が確定する最上階到達時に加算する扱いにした。
  • QRコード生成は外部CDNではなく、MITライセンスのqrcode-generator@1.4.4をリポジトリに同梱する選択にした。会場ネットワークに依存しないため。
  • GAS接続先はランタイム入力ではなくビルド時定数にした。運営中に誤ったURLを入力するリスクを避けるため。
  • GASへのPOSTはapplication/jsonではなくtext/plainでJSON文字列を送る。Apps Script Web AppでCORSプリフライトに詰まりにくい運用を優先した。
  • GASホスト認証は、静的クライアントに埋め込むapiKeyと、運営者が入力するhostPasswordの二段構成にした。認証後のhostTokenはCacheServiceに保存するランダムトークンで、推測可能な自己記述トークンにはしない。
  • apiKeyはデプロイIDと同値であることを正とし、過去セットアップで作られたevg-形式の値がconfigシートに残っていてもデプロイIDを受理する。再度setupElevatorGameSheets()を実行した場合はevg-形式をデプロイIDへ移行する。
  • Hostブラウザ集計commitは、運営用Hostを信頼する設計に寄せる判断をした。不正耐性よりGAS実行時間削減を優先し、最低限の防御としてhostToken、roomVersion、ステージ一致、二重集計拒否を入れる。
  • Apps Scriptエディタに直接配置するファイルはCode.gsだけにし、appsscript.jsonはclasp等で使う任意のmanifestとして扱う。
  • topPlayer予想の正解は、予想イベント得点を加える前のステージ得点1位にした。予想得点によって正解が変わる自己参照を避けるため。
  • 予想イベントでmetricとcorrectAnswerが両方指定された場合は、ゲーム結果から得られるmetricを正解にする。correctAnswerは固定正解や暫定設定用のフォールバックに限定した。
  • 複数プレイヤーブラウザハーネスでは、testSlotクエリをUUID保存キーのサフィックスとして使う。同一ブラウザ内でもプレイヤーごとのUUIDを分離して、実端末を複数用意しなくても参加フローを確認できるようにした。
  • 次ゲームへの移行は、設定JSON importを「参加者保持の新ゲーム開始」として扱うことにした。参加者を破棄する用途は既存のResetに分ける。
  • Playerの中間ランキング保持は端末内メモリのスナップショットで実装した。リロードを伴わない通常操作ではホストが次ステージへ進めてもPlayerタブはランキングのまま残る。
  • GASのplayersシートは現在ゲーム参加者だけで上書きせず、既存UUIDを保持したまま現在参加者をマージする。過去UUID復元の起点を消さないため。

仕様解釈

  • 強制下車に巻き込まれた既存乗客は、強制下車前に人数判定を通過済みの階数分を上昇報酬の対象にする。当該階で新たに乗ろうとして強制下車になったプレイヤーは成功階数0階とする。
  • E3aの区間倍率は、人数判定通過後に乗車中だった成功階がイベントのS〜T内にある場合に適用する。例: 7〜10階は7階、8階、9階、10階の成功階が対象。
  • E3bの得点倍率は、申告乗車階から申告降車階までの経路がS〜Tに触れる場合、乗車成功点全体へ適用する。要件の「条件は別途指定」が未確定のため、設定JSONにあるfromFloor/toFloorを条件にした。
  • E5は、人数判定通過後に乗車中人数がしきい値以上だった成功階の乗車成功点へ倍率を掛ける。
  • E4は、人数判定通過時にその階にいたプレイヤーへ加点する。ブラウザエンジン・GAS集計とも同じ判定にしている。
  • 同一階指定は、人数判定通過後に即時下車し、成功階数1階として扱う。運賃も申告階数1階分を支払う。
  • 範囲選択の予想は、選択肢のvalueを回答として保存し、正解メトリクスの数値が選択肢のmin/max内なら正解とする。片側だけ指定した範囲も許容する。
  • プレイヤー指名の予想は、表示名ではなくUUIDを回答として保存する。名前変更があっても判定対象を安定させるため。
  • 名前重複は現行ルーム内のプレイヤーで判定している。今後GAS接続時は、UUIDと現在名の対応を管理するplayersまたはsave_data由来の現在名で重複判定する。
  • GASのreveal-resultは、すでに当該ステージの結果が存在する場合は拒否する。再集計は別APIのrecalculateに限定し、通常進行で累積得点とSkill履歴が二重加算されないことを優先した。
  • 完了済みゲームのローカル戦歴はcompletedGamesに退避する。次ゲームのゲーム内スコアは0から始めるが、History画面では退避済み結果と現在ゲーム結果を合算して表示する。
  • Playerの結果表示解禁は、スクリーン演出のanimationStartedAtとステージ階数から計算した最終階到達時刻で判定する。ホストのフェーズ遷移ではなく演出完了時刻に合わせる。
  • GASのsave_dataに保存する12指標はゲーム終了時点の事前集計値とした。個別ステージ再計算に必要なStageSkillと結果詳細はstage_resultsへ保存する。
  • GASのPlayer向け状態レスポンスは、UI表示に必要な参加者名と本人チケットを優先し、全チケットや発表前結果は返さない。Screen向け状態は投影表示のため全体チケットと結果を返す。

逸脱

  • GAS通信はUSE_GAS_API有効時のfetch接続、hostToken連携、serverTime補正まで実装した。ただし、実GASデプロイURLでの現地疎通、同時接続負荷、CORS挙動のブラウザ横断確認は未実施。
  • BGM/SEファイルは同梱していない。音声ファイル名規約が未確定で、素材も未提供のため、スクリーンの初回「準備完了」とAudioContext起動のみ実装した。
  • GASのcurrent_gameは完全正規化ではなく、状態JSONを複数行チャンクで保存する実装にした。初版の整合性と競合制御を優先しつつ、1セル上限を避けるため。
  • Skip検知はGAS通信中の結果発表フェーズで低頻度フェッチと最上階到達時の最終確認として実装した。ローカルモードでは同一ブラウザのstorageイベントと再描画で追従する。
  • 同時ホストのボタンロックは、GASのLockServiceによる先着処理を優先し、UI上のリアルタイムロック表示は未実装。
  • Playerの中間ランキング保持はブラウザメモリ上の状態であり、ページリロード後までランキング画面を固定する永続化は未実装。

トレードオフ

  • GASシートを完全正規化する案もあったが、進行中状態はチャンク化JSONスナップショット方式にした。理由は、複雑なステージ設定・投票・演出状態を一括でロック更新でき、同時ホスト競合を扱いやすいため。履歴とマスタはsave_data等へ分割保存する。
  • クライアントでWebSocket相当のリアルタイム通信層まで作る案もあったが、GAS前提ではポーリングを選んだ。参加者100人規模では、要件通りフェーズ別ポーリング間隔の調整とサーバ側キャッシュが次の負荷対策になる。
  • 登録前トップ画面でもポーリングして即時にゲーム進行へ追従する案は、Apps Scriptの実行時間上限に対して費用が大きいため採用しない。参加済み端末、認証済みHost、表示開始済みScreenだけが低頻度で状態取得する。
  • 快適度への影響として、Playerのフェーズ追従は自動即時ではなくなる。投票中の締切検知は最大10秒遅れ、ランキング後の次フェーズ追従は「次へ」押下時確認になる。一方、同一端末ScreenはGAS待ちがなくなり反応が良くなる。
  • 戦績キャッシュは通信量削減を優先し、同一ステージ内では古いGAS戦績を再取得しない。現在進行中ステージのローカルroom由来の表示は引き続き即時反映する。
  • Host自動集計は、認証済みHost画面が開かれていることを前提にした。Host端末が閉じている場合は、別端末Screen/Playerだけでは集計commitを開始しない。
  • イベント仕様をすべて細かいGUIにする案もあったが、初版の設定GUIはJSON import/export中心にした。イベント種別が多く、先にスキーマ互換性を保つ方が安全なため。
  • GAS側へブラウザ版エンジンを完全移植する案もあったが、サイズと検証時間を抑えるために主要ルール中心の実装にした。正式運用前には差分をなくす必要がある。
  • topPlayerを最終得点ベースにする案もあったが、予想イベント得点を含めると循環参照になるため、予想イベント加算前の基礎得点ベースを採用した。
  • ブラウザ複数プレイヤーテストは、まず依存追加なしのHTMLハーネスにした。Playwright等の追加導入よりも、会場運用者がすぐ手動再現できることを優先した。
  • 結果発表のスクロールは、DOMを保持して細かく差分更新する案もあったが、既存の全体再描画構造を保ち、CSSアニメーションの負の遅延で現在時刻に追従させる実装にした。
  • 結果発表完了後は、Screenの再描画を続けると手動スクロール位置が戻るため、最終階到達後の自動再描画を止める設計にした。Skipやホスト操作の反映はフェーズ更新時の再取得で扱う。
  • Playerランキングの次へはサーバに進行要求を送る案もあったが、ホストだけがゲーム全体を進める方針を優先し、クライアント側のポーリング再開操作に変更した。GASの/api/player/proceed-nextもフェーズを進めない確認APIへ変更した。
  • GASのホストセッションはPropertiesServiceではなくCacheServiceに置いた。永続化よりも、期限切れと再認証を明確にする方を優先したため。
